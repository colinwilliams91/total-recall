## Why

Phase 3 wired the intelligence pipeline: diff → concepts → question. But the delivery mechanism is `terminal.Adapter`, which writes to daemon stdout — the terminal where `total-recall serve` is running, not the terminal where the developer committed. Developers who run the daemon in the background (the common case) never see their recall questions.

Phase 4A delivers questions where developers actually are:

1. **MCP** (primary) — coding agents (Claude Code, Copilot CLI) connected to the daemon via the Model Context Protocol receive recall questions as MCP tool results, rendered inline in their agent context.
2. **Shell** (reliable fallback) — a new `tr ask` subcommand, called by a post-commit hook generated by `tr init`, polls the daemon and renders the question interactively in the developer's own git terminal.

After this phase, every developer using Total Recall — whether they work primarily in an AI agent harness or in a plain terminal — has a first-class delivery path. The daemon-stdout fallback is demoted to an opt-in.

## What Changes

- `~/.tr/concepts.db` renamed to `~/.tr/memory.db` — a single SQLite file housing all persistent Total Recall state; a `questions` table is added alongside the existing `concepts` table
- `internal/cache/store.go` updated: DB path, questions schema, and new methods `SaveQuestion`, `NextQuestion` (atomic SQL claim), `AnswerQuestion`
- MCP server wired as first-class: `mcp.NewStreamableHTTPHandler(mcpServer, nil)` mounted at `/mcp/` on the existing `http.ServeMux`; `internal/engine/server.go` gains a `*mcp.Server` field
- MCP tools registered: `recall_next` (atomically dequeue next question), `recall_answer` (record answer or skip), `recall_status` (daemon health + queue depth)
- MCP resources registered: `recall://queue` (subscribable, live pending queue), `recall://recent` (answered/skipped history)
- MCP prompt registered: `recall_workflow` (system prompt snippet guiding the agent to call recall tools after commits)
- Pipeline bridge: after `recall.Synthesize` in `runPipeline`, the question is saved to `memory.db` and `NotifyResourceUpdated("recall://queue")` is called on all connected MCP sessions
- REST delivery endpoints added: `GET /recall/next` and `POST /recall/answer` — serve the shell path without requiring an MCP client
- New subcommand `tr ask`: polls `/recall/next`, displays a "Thinking." → ".." → "..." animation, renders the question with numbered choices, captures keypress [1-3] or Enter to skip, posts the answer, exits cleanly; TTY-aware (CI-safe)
- `tr init` extended: post-commit hook is now generated and installed alongside existing hooks; the hook calls `tr ask`
- `terminal.Adapter` demoted: no longer the default dispatcher; retained as an explicit opt-in behind `presentation.terminal: true` in `~/.tr/config.yaml`

## Key Design Decisions

- **MCP as primary, shell as reliable fallback**: MCP is the first-class delivery path for AI agent harness users; `tr ask` + post-commit hook is the first-class path for vanilla terminal users. Both call the same atomic store methods — neither is a wrapper of the other.
- **Atomic SQL dequeue**: `GET /recall/next` and `recall_next` both execute a single `UPDATE ... WHERE id = (SELECT ...) RETURNING *`. SQLite serialized writes guarantee exactly-once delivery regardless of concurrent callers. No application-level coordination needed.
- **`memory.db` unified store**: all persistent state (`concepts` + `questions`) lives in one file. Future phases (answer scoring, streaks, spaced repetition) add tables, not files.
- **`tr ask` is TTY-aware**: if stdout is not an interactive terminal (CI/CD, piped), `tr ask` exits 0 silently — the post-commit hook never blocks automated pipelines.
- **"Thinking." animation, not a spinner**: cycles `.` → `..` → `...` → `.` at 400ms per frame. Implies background work and encourages the developer to tab away, rather than suggesting they watch for completion.
- **`terminal.Adapter` demoted**: retained as opt-in for `presentation.terminal: true` users (daemon-pane workflow); no longer the default.
- **VS Code extension deferred to Phase 4B**: stub comment in `internal/engine/dispatcher.go` pointing to the Phase 4B scope; no TypeScript in this phase.

## Non-Goals

- VS Code extension implementation (Phase 4B)
- Spaced repetition / answer scoring / difficulty progression (Phase 4B)
- Daemon autostart (Phase 4B)
- Multi-question queuing per commit (future phase — Phase 4A always synthesizes exactly one question per hook event, same as Phase 3)
- Empirical validation of Claude Code / Copilot CLI SSE push rendering behavior (best-effort bonus; REST + `tr ask` is the guaranteed path)
