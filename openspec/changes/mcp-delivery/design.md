## Context

Phase 3 left the daemon in this state: hook fires → 202 Accepted → goroutine → pipeline → `terminal.Adapter.Dispatch(question)` → daemon stdout. Questions have nowhere to go except the background process terminal. This design wires two first-class delivery paths and gives them a shared, atomic backing store.

The central bridge point is `(*Server).runPipeline()` in `internal/engine/server.go`. After `Synthesize` returns a non-nil question, the goroutine saves it to `memory.db` and notifies connected MCP clients. REST endpoints and the `tr ask` subcommand both read from the same store using atomic SQL dequeue.

The MCP server co-exists with existing HTTP routes on port 7331. No new port, no new process. `mcp.NewStreamableHTTPHandler` returns a standard `http.Handler` that mounts directly onto the existing `http.ServeMux`.

## Goals / Non-Goals

**Goals:**
- MCP server (JSON-RPC 2.0 + SSE) mounted at `/mcp/` on port 7331
- MCP tools: `recall_next`, `recall_answer`, `recall_status`
- MCP resources: `recall://queue` (subscribable), `recall://recent`
- MCP prompt: `recall_workflow`
- `GET /recall/next` and `POST /recall/answer` REST endpoints
- `tr ask` subcommand with Bubbletea animation, poll loop, interactive answer, TTY detection
- Post-commit hook generated by `tr init`
- `memory.db` unified store: rename + questions table + atomic dequeue + migration guard
- `terminal.Adapter` demoted to opt-in

**Non-Goals:**
- VS Code extension (Phase 4B)
- Spaced repetition scoring (Phase 4B)
- Daemon autostart (Phase 4B)
- Multi-question queuing per hook event

## Decisions

### 1. Route layout — MCP co-located on port 7331

**Decision**: `mcp.NewStreamableHTTPHandler(mcpServer, nil)` is mounted at `/mcp/` in `RegisterRoutes()`. Two new REST routes are added alongside it.

```
http.ServeMux (port 7331)
├── GET  /health
├── POST /hooks/pre-commit
├── POST /hooks/commit-msg
├── POST /hooks/pre-push
├── GET  /recall/next          ← new: atomic dequeue for tr ask
├── POST /recall/answer        ← new: record answer for tr ask
└── /mcp/                      ← new: mcp.NewStreamableHTTPHandler(mcpServer)
                                   JSON-RPC 2.0 + SSE transport, all handled by SDK
```

`engine.Server` gains a `*mcp.Server` field. MCP server construction and feature registration live in a new file `internal/engine/mcp.go` for separation of concerns. `cmd/total-recall/main.go` constructs the `*mcp.Server` and passes it to `engine.New()`.

---

### 2. Atomic dequeue — exactly-once delivery

**Decision**: Both `/recall/next` and `recall_next` use a single SQL UPDATE with RETURNING:

```sql
UPDATE questions
SET delivered_at = datetime('now'), claimed_by = ?
WHERE id = (
  SELECT id FROM questions
  WHERE delivered_at IS NULL
  ORDER BY queued_at ASC
  LIMIT 1
)
RETURNING id, question, choices, queued_at;
```

The `claimed_by` argument is `"shell"` for REST callers and `"mcp"` for MCP tool callers.

**Rationale**: `tr ask` (post-commit hook) and a connected MCP client (Claude Code) may attempt to dequeue the same question within milliseconds of a commit. SQLite's serialized write semantics guarantee exactly one caller receives the row. The other gets zero rows — an expected, clean "nothing in queue" state. Both callers handle this gracefully: `tr ask` exits 0 silently, `recall_next` returns null. No application-level mutex, channel, or filter needed.

---

### 3. MCP notification bridge from runPipeline

**Decision**: After saving a question to `memory.db`, `runPipeline` notifies connected MCP sessions:

```go
if question != nil {
    if err := s.store.SaveQuestion(ctx, question); err != nil {
        log.Printf("[pipeline] save question: %v", err)
    }
    for sess := range s.mcpServer.Sessions() {
        _ = sess.NotifyResourceUpdated(ctx, "recall://queue")
    }
}
```

The `recall://queue` resource handler reads live from the DB on every call — there is no in-memory queue to keep in sync. `NotifyResourceUpdated` fires the SSE `notifications/resources/updated` event to subscribed clients, which may proactively call `recall_next`. This is best-effort; REST + `tr ask` is the guaranteed delivery path.

### 4. questions table schema

**Decision**:

```sql
CREATE TABLE IF NOT EXISTS questions (
    id           INTEGER  PRIMARY KEY AUTOINCREMENT,
    question     TEXT     NOT NULL,
    choices      TEXT     NOT NULL,   -- JSON array: ["choice1","choice2","choice3"]
    queued_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
    delivered_at DATETIME,            -- set on atomic dequeue (nil = unclaimed)
    claimed_by   TEXT,                -- "mcp" | "shell" | "terminal"
    answer       TEXT,                -- choice text, or "skip"
    answered_at  DATETIME
);
```

`choices` is stored as a JSON array in a TEXT column — avoids a join for what is always a small, bounded list. `claimed_by` is a diagnostic field. `answer` stores choice text rather than an index, so it remains meaningful if choices are reordered in future phases.

### 5. memory.db migration guard

**Decision**: `store.Open()` checks for the legacy `concepts.db` at open time:

```go
memoryPath := filepath.Join(trDir, "memory.db")
legacyPath := filepath.Join(trDir, "concepts.db")

if _, err := os.Stat(memoryPath); os.IsNotExist(err) {
    if _, err := os.Stat(legacyPath); err == nil {
        // copy legacy → memory.db
        if copyErr := copyFile(legacyPath, memoryPath); copyErr == nil {
            log.Printf("[store] migrated %s → %s", legacyPath, memoryPath)
        }
    }
}
```

This is a one-time silent migration. The legacy file is not deleted (power users can remove it manually). No schema version table needed — the `questions` table is added by `CREATE TABLE IF NOT EXISTS` which is idempotent on the migrated file.

### 6. `tr ask` subcommand — Bubbletea design

**Decision**: Cobra command `total-recall ask [--timeout N]` (default timeout: 15 seconds).

State machine:
```
           ┌─────────────────────────────────────────────────┐
           │                 tr ask                          │
           └─────────────────────────────────────────────────┘
                              │
                    TTY check on start
                    (not TTY → exit 0)
                              │
                              ▼
                    ┌──────────────────┐
                    │    "thinking"    │◄──── tick every 400ms
                    │                  │      advance animation frame
                    │  Thinking.       │      poll GET /recall/next
                    │  Thinking..      │
                    │  Thinking...     │
                    └────────┬─────────┘
                             │
               ┌─────────────┴──────────────┐
               │                            │
          no question                  question received
          + timeout elapsed             │
               │                       ▼
               ▼              ┌──────────────────────┐
          exit 0              │      "question"       │
          silently            │                       │
                              │  🧠  Recall Check    │
                              │  ─────────────────   │
                              │  <question text>     │
                              │  1. <choice>         │
                              │  2. <choice>         │
                              │  3. <choice>         │
                              │  ─────────────────   │
                              │  [1-3] or Enter:     │
                              └──────────┬───────────┘
                                         │
                              keypress handler
                                         │
                        ┌────────────────┼────────────────┐
                        │                │                │
                      '1'-'3'          Enter            'q'/Esc
                        │                │                │
                  POST answer        POST skip        exit 0
                  print "✓ recorded" print "→ skipped" (abandon)
                  exit 0             exit 0
```

Animation frames cycle at 400ms. Each tick: advance frame + poll (if no outstanding poll). Polling is not concurrent — the Bubbletea `Cmd` model ensures one poll in flight at a time.

**Daemon unreachable**: if `GET /recall/next` returns a connection error (daemon not running), `tr ask` exits 0 silently with no output. The post-commit hook must never block or error visibly in normal workflows.

---

### 7. post-commit hook

**Decision**: `tr init` generates and `chmod +x`s `.git/hooks/post-commit`:

```sh
#!/bin/sh
# Total Recall post-commit hook
# Surfaces a recall question after each successful commit.
# Generated by 'total-recall init'. Run 'total-recall init' again to reinstall.
exec "$(which total-recall)" ask
```

`exec` replaces the shell process with `tr ask`, keeping process count minimal. `$(which total-recall)` avoids hardcoding the binary path. The hook is idempotent — running `tr init` again overwrites it cleanly.

### 8. MCP surface — tool and resource handlers

**`recall_next` tool**:
- Input: `{}` (no args)
- Calls `store.NextQuestion(ctx)` — atomic dequeue with `claimed_by = "mcp"`
- Returns `{"id": N, "question": "...", "choices": ["...", "...", "..."]}` or `null` if queue empty

**`recall_answer` tool**:
- Input: `{"id": N, "answer": "choice text or 'skip'"}` 
- Calls `store.AnswerQuestion(ctx, id, answer)`
- Returns `{"ok": true}`

**`recall_status` tool**:
- Input: `{}`
- Returns `{"daemon": "ok", "ai_configured": bool, "queue_depth": N}`

**`recall://queue` resource** (subscribable):
- Handler reads `SELECT COUNT(*) FROM questions WHERE delivered_at IS NULL` + first pending question text
- Returns as `application/json` text resource content
- Clients subscribe; `runPipeline` calls `NotifyResourceUpdated` after each `SaveQuestion`

**`recall://recent` resource**:
- Handler reads last 10 rows from `questions` WHERE `answered_at IS NOT NULL`
- Returns as `application/json`

**`recall_workflow` prompt**:
```
After any git commit in this project, call recall_next to check for a pending 
recall question. If a question is returned, present it to the user and record 
their answer with recall_answer. If the queue is empty, continue normally.
```

## Risks / Trade-offs

**[Risk] SSE push to MCP clients is best-effort** — whether Claude Code or Copilot CLI automatically acts on `notifications/resources/updated` depends on the harness. Mitigation: REST + `tr ask` is the guaranteed shell path; MCP push is a bonus for responsive harnesses.

**[Risk] CI/CD post-commit calling `tr ask`** — if the daemon is not running or no TTY, `tr ask` must never block or error. Mitigation: TTY check at startup (exit 0 if not interactive); connection-refused on poll → exit 0 silently.

**[Risk] `memory.db` rename breaks existing installs** — users with Phase 3 installs have `concepts.db`. Mitigation: migration guard in `store.Open()` copies `concepts.db` → `memory.db` on first run with the new binary; idempotent schema migration adds `questions` table.

**[Risk] `go-sdk` API surface changes** — early-lifecycle library. Mitigation: MCP wiring is isolated to `internal/engine/mcp.go`; the SDK is the official spec-tracking implementation. Impact of upstream changes is bounded.
