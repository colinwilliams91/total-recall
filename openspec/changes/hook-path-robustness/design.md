## Context

The post-commit hook today renders via `fmt.Sprintf(postCommitHookScriptTmpl, powerShellPath, shellPath)` where the two `%s` insertions are the same `os.Executable()`-resolved absolute path, one for the PowerShell branch (with `'` escaping) and one for the sh branch (with `"` escaping). The escaping lives in `buildPostCommitHookScript` at `cmd/tr/main.go:113-117`.

The post-commit hook is the ONLY hook that invokes the binary at hook-fire time. The other three hooks (pre-commit, commit-msg, pre-push) are HTTP clients that POST to `http://localhost:7331/hooks/...` — they don't need the binary path at all. Y4 is the smallest phase in the Y-batch: it touches only the post-commit hook template and the `os.Executable()` capture that populates it.

Explore-phase decisions relevant to Y4:

- **Issue 5 Y4 hook PATH strategy:** chosen 5.B (rely on PATH). Y4 strips the `os.Executable()` capture and the `%s` insertions; the hook template becomes a static string. Y4 was originally given options (a) `command -v tr` first with absolute fallback, (b) rely on PATH only, (c) keep current absolute-path-only behavior (status quo). Explore decision: (b).
- **Issue 4 sentinel decision:** `# total-recall managed` stays; the post-commit template's sentinel line is unchanged by Y4.
- **Issue 6 hooks-dir resolution:** this is Y3's territory (`hooks.ResolveHooksDir` + `NewInstaller(repoRoot, hooksDir)`). Y4 doesn't touch the install path; only the template body.
- **Y3 fold-in:** Y3's tasks already include a Section 7 that conditionally applies Y4's changes if the reviewers jointly decide to fold. This Y4 design is written so it stands alone OR folds into Y3 cleanly.

Existing layer model: Y4 touches only the **binary → post-commit hook (path)** leak-point row in `INSTALL_LAYERS.md:41`. That coupling becomes "At `tr repo` time only | NO coupling — the hook template references `tr` by name, resolved at hook-fire time via the shell's PATH". The "Binary → hooks (content)" leak-point row above it (line 40) is unchanged — `# total-recall managed` sentinel and the body strings continue to be baked into the binary, just the post-commit template's *body* no longer contains a baked binary path (it contains the literal string `tr`).

No users exist. No live migration matters.

## Goals / Non-Goals

**Goals:**
- The post-commit hook template at `cmd/tr/main.go:101-111` no longer captures `os.Executable()` and bakes it into the script. Instead, the template is a static string that calls `tr ask` directly via the shell's PATH resolution.
- After rebuilding/relocating the `tr` binary (e.g. `go install ./cmd/tr` placing a new binary at `$GOPATH/bin/tr`), previously-installed post-commit hooks continue to work without requiring `tr repo` re-runs (as long as the new binary is on PATH).
- The `os.Executable()` call in `runRepo()` (formerly `runInit()`) is removed; the "graceful fallback when os.Executable fails" comment block is removed.

**Non-Goals:**
- No `command -v tr` + absolute-path fallback (5.A). Pure PATH reliance (5.B).
- No re-write of the dispatcher hooks (`pre-commit`, `commit-msg`, `pre-push`) — they don't reference the binary. Their bodies are unchanged.
- No PATH detection in the hook script itself. Y5 owns the `tr init` startup detection.
- No version-handshake between installed hooks and the binary — this remains a documented known-limitation (`INSTALL_LAYERS.md` mentions it).
- No change to the install path resolution (Issue 6 / Y3 owns that). Y4 fixes the *template content*, not the install *destination*.

## Decisions

### Decision 1 — Strip `os.Executable()` capture and `%s` insertions; rely on PATH

The new `postCommitHookScriptTmpl` (post-Y4):
```sh
#!/bin/sh
# total-recall managed
# Surfaces a recall question after each successful commit.
# Generated by 'tr repo'. Run 'tr repo' again to reinstall.
if command -v powershell.exe >/dev/null 2>&1; then
    powershell.exe -NoProfile -ExecutionPolicy Bypass -Command "tr ask"
    exit $?
fi

exec tr ask
```

`postCommitScript := postCommitHookScriptTmpl` (the template is now a self-contained constant; no `fmt.Sprintf`). The `buildPostCommitHookScript(selfPath)` helper is deleted (it had no callers other than the one just removed). The `os.Executable()` capture at lines 259-262 and the graceful-fallback block at line 261 (`selfPath = "total-recall"`) are deleted.

**Rationale (Issue 5 Decision 5.B):** Y5's `tr init` startup detection sits between the user and any broken-hook state. If they ignore Y5's warning and the hook fails, the failure is attributable. The "user invokes by absolute path without putting `tr` on PATH and skips `tr init`" edge case is out-of-contract — every documented install flow includes `tr init` first. Protecting against the conjunction is irrational.

**What this changes mechanically:**
- The post-commit hook works whether `tr` is found via `command -v tr` (Unix) or via the shell's normal PATH resolution for `exec tr ask` — no `os.Executable()` lookup, no fallback path.
- Rebuilding the binary via `go install ./cmd/tr` puts a new binary at `$GOPATH/bin/tr`; no `tr repo` re-run needed for the post-commit hook to find the new binary (as long as `$GOPATH/bin` is on PATH — Y5's domain).
- Users invoking `tr` by absolute path without `tr` on PATH get a silently-failing post-commit. Y5's `tr init` startup check warns before they reach that state.

### Decision 2 — Delete `buildPostCommitHookScript` rather than simplify it to a no-arg function

The function exists solely to format two `%s` substitutions. With the substitutions gone, the function becomes `return postCommitHookScriptTmpl` — a passthrough. Inline the constant directly at the call site (`postCommitScript := postCommitHookScriptTmpl`) and delete the function.

**Rationale:** dead-code-lite is dead code. A passthrough function with one caller adds a level of indirection without value. The original escape logic (`strings.ReplaceAll` for PowerShell and shell quoting) was tied to the `%s` insertion — without that, there's nothing to escape and nothing to do.

**Alternative considered and rejected:** keep the function as `buildPostCommitHookScript() string { return postCommitHookScriptTmpl }` — preserves the abstraction seam for future re-introduction of substitutions. Rejected: YAGNI; if path-baking or any other template-substitution need returns in the future, the function can be reintroduced with its original signature modified to match the new requirement.

### Decision 3 — Tests assert on the static template

`cmd/tr/main_test.go`'s existing `buildPostCommitHookScript` tests assert on the rendered template (after `%s` substitution). Post-Y4 they assert on the constant template string directly. The tests become `strings.Contains` or equality checks against the literal `postCommitHookScriptTmpl` content. The exact form of the tests is in tasks.md.

**Rationale:** no behavior to test beyond "the template is the template." The `os.Executable()` capture had zero test coverage (it was wired inline, not unit-tested). Removing it removes an untested code path and replaces it with a constant.

## Risks / Trade-offs

- **Risk:** User runs `tr init` with `tr` not on PATH and ignores Y5's warning. The post-commit hook installs but fails on first commit. The user encounters a confusing `tr: command not found` error in their terminal during `git commit`. → **Mitigation:** Y5's `tr init` startup detection prints a warning with the exact one-line command for the user's detected shell to add `$GOPATH/bin` to PATH. The documented install flow includes `tr init` before `tr repo`. If the user skips `tr init` entirely and runs `tr repo` directly, Y5 can be amended to also run the PATH check at `tr repo` startup (today only `tr init` runs it). Y5's behavior can be strengthened if this becomes a real issue.
- **Trade-off:** the post-commit hook template loses the "graceful fallback to absolute path when the user invokes `tr init`/`tr repo` by absolute path without `tr` on PATH" behaviour that 5.A would have preserved. The trade-off is documented in `INSTALL_LAYERS.md` line 41's leak-point table: the post-commit hook is no longer coupled to the binary's absolute path at install time; it relies on PATH resolution at fire time.
- **Risk:** a future contributor reintroduces path-baking "to make it work without PATH" and we're back to the original bug. → **Mitigation:** the Y4 design.md (this document) and the spec delta for `post-commit-hook` should document the rationale and reference Y5's role as the PATH-detection layer. A future contributor who wants to add path-baking back must first explore the current `tr init` startup detection logic and decide whether to extend it instead.

## Migration Plan

This is a code-only change.

1. Implement Y4 (the template + capture removal) per tasks.md.
2. Build the new binary via `go build -o tr ./cmd/tr` or `go install ./cmd/tr`.
3. Existing installed post-commit hooks (from prior `tr init`/`tr repo` runs) are detected by the installer's sentinel check on the next `tr repo` and overwritten with the new constant template (no `%s` insertions). No manual cleanup needed.
4. Users who do not re-run `tr repo` continue using the old baked-path post-commit; it still works as long as the old binary location remains valid. If they move/rebuild the binary, they need to re-run `tr repo` to install the new template. (No-userbase means this is only relevant to the maintainer, who knows about it.)

**Rollback:** revert the commit. Previous `buildPostCommitHookScript(selfPath)` logic resumes. Existing installed hooks (which stopped referencing the new template) keep their current static-string content — they were already overwritten on the prior forward-implementation. No uninstall path is needed; the next `tr repo` reverts to the prior template.
