# Recall Question Delivery — Architecture

## Overview

Total Recall generates recall questions asynchronously after every Git hook event. The question is synthesized by the AI provider and must reach the developer's terminal. This document describes the Phase 4A delivery paths and the plan for Phase 4B.

## Phase 4A Delivery (Current)

Phase 4A delivers questions where developers actually are, via two first-class paths:

```
git commit  →  pre-commit hook  →  POST /hooks/pre-commit  →  202 Accepted  (immediate)
                                                           ↓
                                                 goroutine: runPipeline()
                                                           ↓
                                        ExtractConcepts → cache.Save → Synthesize
                                                           ↓
                                           store.SaveQuestion (memory.db)
                                                    ↙         ↘
                                       mcpServer.ResourceUpdated   dispatcher (opt-in)
                                             ↓                         ↓
                                      MCP clients                  terminal.Adapter
                                    (recall_next tool)           (presentation.terminal: true)
                                             ↓
                                    GET /recall/next  ←  tr ask (post-commit hook)
```

### Path 1: MCP (primary, for AI agent harness users)

`mcp.NewStreamableHTTPHandler` is mounted at `/mcp/` on port 7331. AI coding agents (Claude Code, Copilot CLI) connected via MCP receive questions through the `recall_next` tool. After `Synthesize`, `runPipeline` calls `mcpServer.ResourceUpdated("recall://queue")` to notify subscribed clients.

**MCP surface:**
- `recall_next` tool — atomically dequeues one question (`claimed_by = "mcp"`)
- `recall_answer` tool — records answer or `"skip"`
- `recall_status` tool — daemon health, `ai_configured`, `queue_depth`
- `recall://queue` resource — subscribable; updated after each `SaveQuestion`
- `recall://recent` resource — last 10 answered questions
- `recall_workflow` prompt — system prompt instructing the agent to call `recall_next` after commits

### Path 2: Shell (`tr ask`, for terminal users)

A post-commit hook generated by `tr init` calls `total-recall ask` after each successful commit. `tr ask` polls `GET /recall/next`, displays a "Thinking." animation, and renders the question interactively.

```
GET /recall/next  →  200 OK (question JSON) or 204 No Content
POST /recall/answer  →  {"ok":true}
```

Both REST endpoints use the same atomic `UPDATE ... RETURNING` dequeue as `recall_next`. SQLite's serialized writes guarantee exactly-once delivery.

`tr ask` is TTY-aware: if stdout is not an interactive terminal (CI/CD, piped output), it exits 0 silently.

### Unified backing store: `~/.tr/memory.db`

All persistent state lives in a single SQLite file. The `questions` table supports both delivery paths:

```sql
CREATE TABLE questions (
    id           INTEGER  PRIMARY KEY AUTOINCREMENT,
    question     TEXT     NOT NULL,
    choices      TEXT     NOT NULL,  -- JSON array
    queued_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
    delivered_at DATETIME,           -- set on atomic dequeue
    claimed_by   TEXT,               -- "mcp" | "shell" | "terminal"
    answer       TEXT,
    answered_at  DATETIME
);
```

`memory.db` is renamed from the Phase 3 `concepts.db`. `store.Open()` includes a one-time migration guard that copies `concepts.db` → `memory.db` when `memory.db` does not yet exist.

## terminal.Adapter (opt-in)

`terminal.Adapter` is retained as an opt-in dispatcher for users who prefer the daemon-pane workflow:

```yaml
# ~/.tr/config.yaml
presentation:
  terminal: true  # enables terminal.Adapter dispatcher
```

When `presentation.terminal: false` (the default), the dispatcher is nil and questions are delivered only through MCP and REST.
